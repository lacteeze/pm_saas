    This is the most critical migration. Three steps plus Auth Hook update in one file:

    Step 1 — Drop old check constraint: ALTER TABLE public.people DROP CONSTRAINT IF EXISTS people_role_check;

    Step 2 — Cast column: ALTER TABLE public.people ALTER COLUMN role TYPE TEXT[] USING ARRAY[role];

    Step 4 — Replace Auth Hook (per D-06 and Option A from RESEARCH.md): CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event jsonb) RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER SET search_path = '' AS $$ body with: SELECT p.id, p.org_id, p.role INTO person_record FROM public.people p WHERE p.user_id = (event->>'user_id')::uuid LIMIT 1; then jsonb_set for org_id, role (person_record.role[1] — the first element as text, not the array), person_id. This emits a string JWT role claim, not an array — all existing RLS policies remain valid with no changes $$. Include a comment: "Emits first role element as JWT claim string. role text[] column is used by UI queries; JWT claim stays a single string so RLS helper public.user_role() works unchanged."

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Auth Hook → JWT | role[1] extracted from text[] array; if array is empty, claim is null — handled by IS NOT NULL check at login |
| Storage path → lease ownership | Path segment [3] used as lease_id; malformed paths won't match any lease row |
| Tenant storage policy → lease data | EXISTS subquery verifies tenant_id matches person_id — cross-tenant path crafting is blocked |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | people.role migration | mitigate | ARRAY[role] cast preserves existing role values; new CHECK constraint ensures at least one valid role; no way to escalate via migration |
| T-02-02 | Information Disclosure | leases_select_tenant | mitigate | RLS policy binds to (SELECT public.person_id()) — tenant can only see their own lease row |
| T-02-03 | Information Disclosure | storage tenant lease access | mitigate | Storage policy + EXISTS subquery on leases table verifies lease belongs to requesting tenant |
| T-02-04 | Tampering | leases INSERT by non-manager | mitigate | leases_insert_manager WITH CHECK verifies user_role() IN ('manager', 'admin') |
| T-02-05 | Elevation of Privilege | Auth Hook role[1] | mitigate | Hook emits first array element; if a tenant had 'manager' injected as second role, only their first role (e.g. 'tenant') is emitted in JWT — UI role picker must not allow 'admin' role for contacts |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No new packages in Phase 2; this plan has zero install tasks |
</threat_model>
