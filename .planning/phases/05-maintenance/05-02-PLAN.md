<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server Action inputs → DB writes | All status transitions validated against TRANSITIONS map before any DB write |
| vendor_token (unauthenticated) → work order mutation | Admin client used; token validated by DB lookup not session |
| Owner approval token → status change | Tokens nullified in same DB write as status update |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Tampering | updateWorkOrderStatus — $500 gate bypass | mitigate | All status mutations MUST go through updateWorkOrderStatus; no direct `.update({ status })` elsewhere; validateTransition called before any write |
| T-05-05 | Spoofing | Owner approval tokens reuse | mitigate | owner_approve_token + owner_decline_token nullified atomically with status update; second request finds no matching row |
| T-05-06 | Information Disclosure | vendor_cost / billed_amount exposed to tenants | mitigate | createWorkOrder and tenant-facing queries must never SELECT * — explicitly exclude cost columns |
| T-05-07 | Tampering | updateViaVendorToken accepts arbitrary status string | mitigate | Parameter type restricted to 'in_progress' | 'completed'; additional state validation inside function |
| T-05-08 | Information Disclosure | PINGRAM_API_KEY in server env | accept | Key used only in server actions / lib; never in NEXT_PUBLIC_ namespace |
| T-05-SC | Tampering | npm/pingram install | mitigate | pingram v1.0.14 confirmed on npm; Approved in RESEARCH.md Package Legitimacy Audit |
</threat_model>
